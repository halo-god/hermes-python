# 运维文档

面向部署、运行、监控本系统的运维 / SRE。

---

## 1. 部署架构

```
            ┌─ web (nginx)  静态前端 + 反代 /api /ws → api
浏览器 ───→ │
            └─ api (FastAPI/uvicorn)  REST + SSE + WebSocket
                  │            │
                  ▼            ▼
            PostgreSQL      Redis ──(Stream/PubSub)── agent-runner ──spawn──> ACP 子进程
                  │                                        │
                  ▼                                        ▼
              业务主数据                              工作区产物（DB 或 MinIO）
```

| 服务 | 角色 | 有状态 | 扩展 |
|---|---|---|---|
| `web` | nginx：前端静态 + `/api`·`/ws` 反代 | 否 | 多副本 |
| `api` | REST + SSE + WebSocket | 否 | 多副本（SSE/WS 经 Redis PubSub 跨实例） |
| `agent-runner` | ACP 网关：扫描 + spawn agent 子进程 | 偏有状态（本地进程/磁盘） | 池化 / 专用节点 |
| `postgres` | 业务主库 | 是 | 主从 / 托管 RDS |
| `redis` | 缓存 / 队列 / PubSub / 限流 | 是（AOF） | Sentinel / Cluster |
| `minio` | 对象存储（可选） | 是 | 分布式 / 云 S3 |

## 2. Docker Compose 部署

```bash
cp .env.example .env          # 按 §4 配置
make up                       # = docker compose -f docker/compose.yaml --env-file .env up --build -d
```

端口：Web `8080`、API `8000`、MinIO `9000`/控制台 `9001`、PG `5432`、Redis `6379`。
数据卷：`pgdata`、`redisdata`、`miniodata`、`workspacedata`。

容器启动顺序由 healthcheck 把控：`api` 等 PG/Redis 健康；入口脚本 `scripts/start.sh` 自动 `alembic upgrade head` + `python -m app.seed` 再起 uvicorn。`agent-runner` 用 `scripts/run-agent.sh`。

## 5. 数据库迁移

```bash
# 容器内
docker compose -f docker/compose.yaml exec api alembic upgrade head
docker compose -f docker/compose.yaml exec api alembic current
docker compose -f docker/compose.yaml exec api alembic downgrade -1   # 回滚一格
```
迁移文件 `backend/alembic/versions/0001…0006`，线性串联。**发布顺序**：先停写 → 备份 → `upgrade head` → 起新版本。迁移都写了 `downgrade`，但生产回滚务必先验证数据兼容。

## 6. 备份与恢复

**PostgreSQL（主数据，最重要）**
```bash
# 备份
docker compose -f docker/compose.yaml exec -T postgres pg_dump -U hermes hermes | gzip > hermes_$(date +%F).sql.gz
# 恢复
gunzip -c hermes_2026-05-30.sql.gz | docker compose -f docker/compose.yaml exec -T postgres psql -U hermes hermes
```
建议每日全量 + WAL 归档（或托管 RDS 自动备份）。

**Redis**：开启 AOF（compose 已 `--appendonly yes`）。Redis 内容多为可重建的缓存/计数；关键的是 PubSub 即时数据无需备份。备份 `redisdata` 卷即可。

**MinIO / 工作区**：备份 `miniodata` 卷或用 `mc mirror` 到异地；`STORAGE_BACKEND=db` 时工作区随 PG 一起备份。

**配置/密钥**：`.env` 与密钥单独安全保管（勿入库）。

## 7. 可观测性

### 7.1 健康检查
- `GET /api/v1/healthz` — 存活（liveness）。
- `GET /api/v1/readyz` — 就绪：检查 PostgreSQL + Redis，返回 `{ready, checks}`。
- 容器 healthcheck 已配置；K8s 用作 liveness/readiness probe。

### 7.2 Prometheus 指标
`GET /metrics`（Prometheus 文本格式）。核心指标：

| 指标 | 类型 | 标签 | 含义 |
|---|---|---|---|
| `hermes_http_requests_total` | Counter | method,path,status | HTTP 请求数 |
| `hermes_http_request_seconds` | Histogram | method,path | 请求延迟分布 |
| `hermes_logins_total` | Counter | result(ok/fail) | 登录次数 |
| `hermes_messages_total` | Counter | — | 消息发送数 |

Prometheus 抓取示例：
```yaml
scrape_configs:
  - job_name: hermes-api
    metrics_path: /metrics
    static_configs: [{ targets: ['api:8000'] }]
```
建议告警：5xx 比率、p99 延迟、`hermes_logins_total{result="fail"}` 突增（暴力破解）、readyz 失败。

### 7.3 日志
结构化文本日志到 stdout（`app/core/logging.py`），`DEBUG=true` 时含 agent 子进程 stderr 逐行。生产用 Loki/ELK 采集 stdout。关键 logger：`hermes`、`hermes.runner`、`hermes.acp`、`hermes.discovery`、`hermes.sandbox`。

### 7.4 审计
安全相关动作写 `audit_log` 表（登录成功/失败、管理员改用户/设置/身份）。后台「审计日志」或 `GET /api/v1/admin/audit` 查询。

## 8. 扩缩容

- **`api`**：无状态，水平扩。SSE/WebSocket 跨实例靠 Redis PubSub，任意实例都能推到对应浏览器。
- **`agent-runner`**：每实例一个 Redis Stream 消费者（`ACP_CONSUMER` 必须各不相同），消费组自动分摊任务；每实例设并发上限（受 CPU/内存约束，agent 子进程是重资源）。建议独立节点池 + 沙箱运行时。
- **`postgres`**：读多可加只读副本；连接数用 `pool_pre_ping` 已开，必要时前置 PgBouncer。
- **`redis`**：Sentinel 高可用或 Cluster 分片。

## 9. 升级流程

1. 备份 PG。
2. 拉新镜像 / 代码。
3. 灰度：先升 `api` 一个副本，观察 `readyz` + 指标。
4. 跑迁移（`alembic upgrade head`，作为发布前置步骤）。
5. 全量滚动更新 `api` / `web` / `agent-runner`。
6. 冒烟：登录 → 发消息 → 看 `/metrics` 计数增长。
7. 异常则回滚镜像（迁移回滚需谨慎，见 §5）。

## 10. 故障排查

| 现象 | 排查 / 处理 |
|---|---|
| `readyz` postgres error | DB 不可达 / 连接数耗尽 / 迁移未跑。看 `DATABASE_URL`、PG 日志、`alembic current` |
| `readyz` redis error | Redis 不可达；看 `REDIS_URL`、Redis 进程 |
| 发消息无响应、无流 | agent-runner 没起 / 没注册 agent。看 runner 日志「Registered N agent(s)」；`GET /api/v1/agents` |
| runner 日志 `subprocess closed — stderr: …` | agent 子进程崩了，stderr 末尾已附上；多为真实 CLI 未鉴权/缺 Key（见[真机联调](真机联调.md)） |
| 登录 429 频繁 | 限流过严；后台「系统设置」调 `rate_limit_per_min`（即时生效） |
| SSE 经 nginx 不流式 | 确认反代关闭缓冲（响应已带 `X-Accel-Buffering: no`；nginx `proxy_buffering off`） |
| WebSocket 连不上 | nginx 需 `Upgrade`/`Connection` 头 + `proxy_read_timeout` 拉长（compose 内 `web` 已配） |
| LDAP 登录 503 | LDAP 不可达/配置错（已转 503 不 500）；核对 `/admin/identity` 的 host/`user_dn_template` |
| 前端 401 后被踢登录 | refresh 失效；正常安全行为。检查时钟/`SECRET_KEY` 是否一致（多副本须同一密钥） |

## 11. 灾备（DR）

- **RPO**：取决于 PG 备份频率（建议 ≤ 1h，WAL 归档可近实时）。
- **RTO**：恢复 = 起 PG/Redis/MinIO → 还原 PG dump → 起 api/runner → 冒烟。
- 定期演练恢复；密钥与 `.env` 异地保管；MinIO 跨区镜像。

---

## 12. Kubernetes（规模化指引）

- `api`/`web` 用 Deployment + HPA + readiness/liveness probe（`/api/v1/readyz` `/healthz`）。
- `agent-runner` 用 StatefulSet 或专用节点池；结合 **gVisor RuntimeClass** 跑 agent 子进程，配 NetworkPolicy 限制出口。
- PG 用云托管或 Patroni；Redis 用 Sentinel/Cluster；对象存储用云 S3。
- 迁移作为发布前置 Job；密钥用 Secret/KMS。
- 详细清单可在此架构上扩展（本仓库默认交付 Compose，K8s 为推荐演进路径）。
