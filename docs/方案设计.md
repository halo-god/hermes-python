# Hermes 信使 — 生产级全栈架构方案

> 把现有 Vue 3 前端原型升级为可在**生产环境**运行的前后端完整系统。
> 技术栈：**FastAPI**（后端） · **PostgreSQL**（主库） · **Redis**（缓存/队列/实时） · **ACP** 接入本机 agent。
> 认证：**本地账号优先**，LDAP/AD 与企业微信 SSO 预留接口、后续阶段对接。
> 本文为**第一阶段交付物：完整方案文档**，确认后再分阶段写代码。

> ### ✅ 已确认决策（2026-05-30）
> | 决策点 | 选择 |
> |---|---|
> | Agent 运行位置 | **服务端集中托管 + 沙箱**（本方案 §1.1 / §12） |
> | 首批支持 agent | **Hermes Agent CLI**（其它 CLI 后续按 §2.2 扩展） |
> | 前端 | **Vite + Vue 3 SFC + TypeScript + Pinia + Vue Router** 工程化 |
> | 部署 | **Docker Compose 起步**（规模化再演进 K8s） |

> | 流式传输 | **SSE**（性能优先：热路径仅 Redis Pub/Sub，不碰 DB；nginx 关闭缓冲、立即 flush）。双向需求（圆桌）后续上 WebSocket |

## 1. 总体架构

### 1.1 关键架构决策：Web 化带来的 agent 运行位置问题

你参考的 **AionUi 是本地桌面应用（Electron）**：agent CLI 跑在**用户自己的电脑**上，扫描的是**本机 PATH**，访问的是**本机文件系统**。

但你要做的是**中心化 Web 应用**：浏览器里没有文件系统、不能 spawn 进程。因此 agent 必须跑在**服务端**。这带来三个必须解决的问题：

1. **谁来扫描 agent？** —— 不再是「用户主机」，而是**专门的 Agent Runner 工作节点**扫描它自己容器内的 PATH。
2. **多用户隔离** —— agent 能执行代码、读写文件、联网，多个用户/团队的会话**绝不能共享同一个文件系统或进程**。必须**每会话沙箱**。
3. **凭证管理** —— `claude login` / API key 这类 agent 自身的认证，要在服务端按租户/团队维度安全托管。

> 这是整个方案与 AionUi 最大的不同点。下文 §2、§12 专门处理。

**进程职责拆分（都是 FastAPI 应用 / Python 服务）：**

| 服务 | 职责 | 扩展方式 |
|---|---|---|
| `api`（FastAPI） | REST 接口、鉴权、业务 CRUD、编排 | 无状态横向扩展 |
| `realtime`（FastAPI + WebSocket/SSE） | 向浏览器推流式 token、tool 事件、圆桌进度 | 横向扩展 + Redis Pub/Sub 跨实例 |
| `worker`（Celery/Arq） | 异步任务：发邮件、SSO 同步、审计落库、定时任务 | 按队列扩展 |
| **`agent-runner`** | **ACP 网关**：扫描可用 agent、按会话 spawn 沙箱子进程、桥接 ACP↔Redis | 池化，每节点容量受限 |

> 小规模起步时 `api`/`realtime`/`worker` 可合并为一个进程；`agent-runner` 因为要 spawn 进程、做沙箱，**建议从一开始就独立部署**。

## 2. Agent 接入层（ACP）—— 核心

这是整个系统的灵魂，也是你前端早已埋好伏笔的地方（`acp-xxxx` session id、Profile 选择器、工作区产出文件、圆桌多 agent）。

### 2.1 ACP 协议速览

**ACP（Agent Client Protocol）** = 编辑器/客户端 与 编码 agent 之间的标准协议，类似 "AI agent 界的 LSP"。
- **传输**：JSON-RPC 2.0，**newline-delimited JSON over stdio**（本地子进程）；远程可走 HTTP/WebSocket。
- **当前稳定版本**：protocol version `1`，在 `initialize` 时协商。

**生命周期（Client = 我们的 agent-runner，Agent = claude/gemini/…）：**

```
runner ──initialize(protocolVersion, clientCapabilities)──────────────> agent
runner <──result(protocolVersion, agentCapabilities, authMethods)────── agent
runner ──authenticate(methodId)──────────────────────────────────────> agent   (若需要)
runner ──session/new(cwd, mcpServers)────────────────────────────────> agent
runner <──result(sessionId)──────────────────────────────────────────── agent
runner ──session/prompt(sessionId, prompt[])─────────────────────────> agent
        <──session/update(agent_message_chunk)──── (流式 token)
        <──session/update(agent_thought_chunk)──── (思考)
        <──session/update(tool_call / tool_call_update)──── (工具调用)
        <──session/update(plan)──── (计划)
        <──fs/write_text_file(path, content)──── (产出文件! → 工作区)
        <──session/request_permission(...)──── (需用户授权)
runner <──result(stopReason: end_turn|max_tokens|cancelled)──────────── agent
runner ──session/cancel(sessionId)───────────────────────────────────> agent   (中断)
```

**关键方法映射到前端已有概念：**

| ACP 方法 / 事件 | 前端对应 |
|---|---|
| `session/new` → `sessionId` | 顶部连接条 `ACP · 主信使 · acp-xxxxx` |
| `session/prompt` | 用户在 composer 发送消息 |
| `session/update: agent_message_chunk` | 聊天气泡流式打字 + Markdown 渲染 |
| `session/update: tool_call` | （可新增）消息内工具调用卡片 |
| `fs/write_text_file` | **工作区面板的 AI 产出文件**（docx/md/csv/json） |
| 多个 `session/prompt` 并行 | **圆桌**多 agent 并行作答 |
| `session/cancel` | 停止生成按钮 |

### 2.2 Agent 探测（扫描主机内 agent）

复刻 AionUi 的做法，但作用域是 **agent-runner 容器自己的环境**：

```python
# agent_runner/discovery.py（示意）
KNOWN_AGENTS = [
    # bin 名,        显示名,          ACP 启动参数
    ("claude",       "Claude Code",   ["--acp"]),          # 实际以各 CLI 文档为准
    ("gemini",       "Gemini CLI",    ["--experimental-acp"]),
    ("qwen",         "Qwen Code",     ["acp"]),
    ("codex",        "Codex",         ["acp"]),
    ("openclaw",     "OpenClaw",      ["acp"]),
    ("cursor-agent", "Cursor Agent",  ["acp"]),
    ("goose",        "Goose",         ["acp"]),
    # … 20+
]

async def scan() -> list[DiscoveredAgent]:
    found = []
    for bin_name, label, acp_args in KNOWN_AGENTS:
        path = shutil.which(bin_name)              # 扫描 PATH
        if not path:
            continue
        version = await _probe_version(path)        # 异步批量，避免串行 execSync 慢
        # 探测 ACP 能力：尝试 initialize 握手，读 agentCapabilities
        caps = await _probe_acp(path, acp_args)
        found.append(DiscoveredAgent(
            bin=bin_name, path=path, label=label,
            version=version, acp_args=acp_args, capabilities=caps,
        ))
    return found
```

- **启动时**：每个 agent-runner 节点 `scan()` 一次，把可用 agent 注册到 **Redis**（`agents:available` 哈希，含节点归属、能力、版本），并写一份快照到 PostgreSQL `agent_runtime` 表供审计。
- **前端 Profile/Agent 选择器**：改为从 `GET /api/agents` 读注册表 —— 真正「主机里装了什么就显示什么」。
- **定期重扫**：worker 定时（或收到 SIGHUP）重扫，支持热插拔新装的 CLI。

> AionUi 的性能优化经验直接采纳：**CLI 探测从串行 `execSync` 改为异步批量执行**，启动连接显著加速；Windows 上覆盖 `TMP`/`TEMP` 让 bunx 工作目录避开杀软扫描路径。

### 2.3 会话 spawn 与桥接

```
用户发消息
  → api 写入 message(pending) → 投递任务到 Redis Stream `acp:prompt`
  → agent-runner 消费：
      1. 取/建该会话的 ACP 子进程（沙箱容器内 spawn `claude --acp`）
      2. 若新会话：initialize → (authenticate) → session/new(cwd=该会话工作目录)
      3. session/prompt(用户内容)
      4. 监听 stdout 的 session/update 流：
           - agent_message_chunk → publish 到 Redis Pub/Sub `chan:session:{id}`
           - fs/write_text_file   → 落对象存储 + 写 workspace_file 表 + 推事件
           - tool_call            → 推事件
      5. stopReason 收到 → 标记 message 完成、token 计费、审计
  → realtime 服务订阅 `chan:session:{id}`，经 WebSocket/SSE 推给浏览器
```

**进程生命周期策略：**
- **一个 ACP 子进程对应一个会话**（保留上下文）；空闲超时（如 30 min）回收，下次用 `session/load` 恢复（若 agent 支持 `loadSession` 能力，否则重建）。
- 每个 runner 节点设**并发上限**（如 N=20 个活跃 agent 进程），超出走排队（Redis）或调度到其它节点。
- **凭证注入**：agent 自身的 auth（`claude login` / `API_KEY`）以租户/团队维度存在 Vault/加密表，spawn 时按需注入到子进程环境，**绝不落明文日志**。

### 2.4 内置 agent（兜底）

主机上未必装了所有 agent。前端 13 个内置 agent（Hermes、Cowork、表单助手…）映射为：
- **优先**：匹配到的本机 ACP agent（如 OpenClaw、Hermes Agent CLI）。
- **兜底**：直连 LLM 厂商 API 的「内置编排器」—— 一个我们自己实现的轻量 ACP-兼容服务（profile→system prompt、工具调用、文件产出），保证没有任何 CLI 时系统仍可用。

> 这样「扫描主机 agent」与「LLM API 兜底」并存：装了 CLI 就用 CLI，没装就用内置编排器，前端无感。

## 3. 技术栈选型

| 层 | 选型 | 理由 |
|---|---|---|
| Web 框架 | **FastAPI** | 异步原生，最适合 SSE/WebSocket 流式与 ACP 长连接；Pydantic v2 强类型；自动 OpenAPI |
| ASGI 服务器 | **Uvicorn**（+ Gunicorn 管理多 worker） | 生产标准组合 |
| ORM | **SQLAlchemy 2.0**（async）+ **Alembic** | 成熟、异步、迁移完善 |
| 校验/序列化 | **Pydantic v2** | 与 FastAPI 一体 |
| 主库 | **PostgreSQL 16** | 事务、JSONB（存 message 富结构）、`pgvector`（后续 RAG 知识库） |
| 缓存/队列/实时 | **Redis 7** | 缓存 + Pub/Sub（推流）+ Stream（任务）+ 限流 + 在线状态 |
| 异步任务 | **Arq**（轻）或 **Celery**（重） | 邮件、SSO 同步、定时任务、审计落库 |
| 对象存储 | **MinIO**（自托管 S3）/ 云 S3 | 工作区产出文件、上传附件 |
| 鉴权 | **JWT**（access+refresh）+ Redis 黑名单 | 本地账号优先；SSO 预留 |
| 沙箱 | **gVisor / Firecracker / 容器 + seccomp** | agent 子进程隔离（§12） |
| 部署 | **Docker Compose**（中小）→ **K8s**（规模化） | 平滑演进 |
| 可观测 | **OpenTelemetry + Prometheus + Grafana + Loki** | 指标/日志/链路 |

### 5.1 ER 概要

```

  │                          │          │
```

### 5.2 关键表定义（简化 DDL）

```sql
-- 用户：本地账号优先，SSO 字段预留
CREATE TABLE users (
  id            UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  email         CITEXT UNIQUE NOT NULL,
  handle        CITEXT UNIQUE,
  name          TEXT NOT NULL,
  initials      TEXT, color TEXT,
  title         TEXT, department TEXT, phone TEXT, timezone TEXT, bio TEXT,
  password_hash TEXT,                       -- 本地账号；SSO-only 用户为 NULL
  source        TEXT NOT NULL DEFAULT 'local',  -- local | ldap | wecom | saml | oidc
  external_id   TEXT,                        -- SSO 唯一标识（预留）
  role          TEXT NOT NULL DEFAULT 'member', -- super_admin|admin|team_admin|member|viewer
  status        TEXT NOT NULL DEFAULT 'active', -- active|pending|inactive
  twofa_secret  TEXT,
  created_at    TIMESTAMPTZ DEFAULT now(),
  last_active_at TIMESTAMPTZ,
  UNIQUE (source, external_id)
);

CREATE TABLE teams (
  id        UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  name      TEXT NOT NULL, handle CITEXT UNIQUE,
  tagline   TEXT, color TEXT, plan TEXT, join_mode TEXT,
  policy    JSONB NOT NULL DEFAULT '{}',     -- 内容权限矩阵（perm×role）
  created_at TIMESTAMPTZ DEFAULT now()
);

CREATE TABLE team_members (
  team_id UUID REFERENCES teams ON DELETE CASCADE,
  user_id UUID REFERENCES users ON DELETE CASCADE,
  role    TEXT NOT NULL DEFAULT 'member',     -- owner|admin|member|viewer
  status  TEXT DEFAULT 'offline',             -- online|idle|offline
  joined_at TIMESTAMPTZ DEFAULT now(),
  PRIMARY KEY (team_id, user_id)
);

CREATE TABLE projects (
  id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  team_id UUID REFERENCES teams ON DELETE CASCADE,
  name TEXT NOT NULL, handle CITEXT, color TEXT, icon TEXT,
  summary TEXT, progress INT DEFAULT 0,
  status TEXT DEFAULT 'active',               -- active|paused|archived
```

## 6. Redis 用途

| 用途 | 数据结构 | 说明 |
|---|---|---|
| 流式推送 | **Pub/Sub** `chan:session:{id}` | agent-runner 发布 token/事件，realtime 订阅转 WS/SSE，**跨实例** |
| 提示词任务队列 | **Stream** `acp:prompt` | api → agent-runner，可靠消费、重试、消费组 |
| Agent 注册表 | **Hash** `agents:available` | 探测结果，前端读取 |
| 会话进程映射 | **Hash** `acp:sessions` | sessionId → runner 节点 / pid，路由用 |
| 用户在线状态 | **Sorted Set** `presence` | 团队成员 online/idle/offline + 心跳过期 |
| 缓存 | String/Hash + TTL | 用户资料、团队元数据、权限矩阵结果（写时失效） |
| 限流 | 令牌桶（Lua 脚本） | 按用户/IP 限频，模型配额（对应后台「速率限制/月度配额」） |
| JWT 黑名单 | Set + TTL | 登出/踢设备（对应用户中心「撤销其它设备」） |
| 分布式锁 | `SET NX PX` | 会话 spawn、定时任务防重 |

## 7. API 设计

REST 走 `//api/v1`，实时走 WS/SSE。下面列**与前端功能一一对应**的核心端点（非全量）。

### 7.1 认证
```
POST /api/v1/auth/login            # {method: local|ldap|wecom, ...} → access+refresh
POST /api/v1/auth/refresh
POST /api/v1/auth/logout
GET  /api/v1/auth/me
GET  /api/v1/auth/providers        # 登录页 tab：哪些 SSO 已启用
# 二阶段：
POST /api/v1/auth/ldap/bind        # LDAP/AD 校验
GET  /api/v1/auth/wecom/qrcode     # 企业微信扫码 → 轮询/回调
```

### 7.2 会话与消息（核心，含流式）
```
GET    /api/v1/conversations                 # 历史目录（搜索/过滤/分组/排序）
POST   /api/v1/conversations                 # 新建会话（选 profile/agents/上下文）
GET    /api/v1/conversations/{id}
PATCH  /api/v1/conversations/{id}            # 改名/置顶/可见性
DELETE /api/v1/conversations/{id}            # 删除（历史页批量删 → DELETE 批量）
POST   /api/v1/conversations/{id}/messages   # 发送 → 触发 ACP；返回 message_id
POST   /api/v1/conversations/{id}/cancel     # 停止生成 → session/cancel
# 流式（二选一）：
GET    /api/v1/conversations/{id}/stream     # SSE：agent_message_chunk / tool / file
WS     /ws/conversations/{id}                # 双向：更适合圆桌、多事件
```


## 8. 认证与 RBAC

### 8.1 第一阶段：本地账号
- 密码 **argon2id** 哈希；登录签发 **JWT access（15min）+ refresh（7d）**；refresh 轮换 + Redis 黑名单。
- 2FA（TOTP）对应用户中心「安全」tab。
- 设备会话列表 + 「撤销其它设备」= refresh token 家族管理（Redis）。

### 8.2 双层权限模型
前端已有两套权限，后端如实落地：

1. **平台级 RBAC**（后台管理）：`super_admin / admin / team_admin / member / viewer` —— 决定能否进后台、管用户等。`core/rbac.py` 用装饰器/依赖 `require_role(...)` 守卫。

### 8.3 二阶段：SSO（预留接口已就位）
- `auth_providers/base.py` 定义 `authenticate()`、`sync_users()`、`resolve_dept_mapping()` 抽象。
- **LDAP/AD**：`python-ldap`/`ldap3` 绑定校验 + 属性拉取；按 `dept_team_mappings` 自动分配角色 / 入团队。
- **企业微信**：OAuth 扫码 + 通讯录同步（worker 定时 `sso_sync`）。
- 前端登录页的 tab、后台身份连接页、部门映射表**已经做好**，二阶段只接后端。

## 9. 实时通信

- **传输选择**：聊天流式用 **WebSocket**（圆桌多事件、可中断、可双向）；若想更省心也可 SSE + `POST cancel`。前端 composer 已有发送/停止交互骨架。
- **跨实例广播**：realtime 实例订阅 Redis Pub/Sub，任意 runner 产出的事件都能推给正确的浏览器连接。
- **圆桌**：api 对 `active_agent_ids` 里每个 agent 各投一个 `acp:prompt` 任务 → 多个 ACP 进程并行 → 各自 `roundtable` 事件帧带 `agentId` → 全部 `done` 后，内置编排器再发一轮「综合」prompt → `merged` 帧。完全复刻前端圆桌 UI。
- **断线重连**：消息已落 PG，重连后用 `GET /conversations/{id}` 拉全量 + 从 `last_event_id` 续订。

## 10. 文件 / 工作区

- agent 通过 ACP `fs/write_text_file`（或在其沙箱 cwd 产出文件）→ runner 捕获 → 上传**对象存储（MinIO/S3）** → 写 `workspace_files` + 版本表 → 推 `file` 事件。

- 上传附件（composer 回形针菜单）走 `POST /files`，注入会话上下文。

## 11. 前端改造（原型 → API 驱动）

现有前端是**纯静态 + 内存 mock**（`hermes-data.js`）。改造点：

| 现状 | 改为 |
|---|---|
| `JSON.parse(JSON.stringify(CONVERSATIONS))` 内存数据 | 调 `GET /api/v1/...`，引入 **Pinia** 做状态管理 |
| 模拟回复 `setTimeout` 打字 | 订阅 **WS/SSE** 真实流式 token |
| 假的 `acp-xxxx` session id | 真实 ACP `sessionId` |
| Profile 选择器写死 | `GET /agents` 动态注册表 |
| 登录页 demo 跳过 | 真实 `POST /auth/login` + JWT 存储 + 路由守卫 |
| 各种 `alert('演示')` / toast | 真实接口 + 乐观更新 + 错误处理 |
| 工作区文件写死 | `GET /conversations/{id}/files` + 签名 URL |

- **构建**：原型是无构建步骤的全局 Vue。生产建议迁到 **Vite + Vue 3 SFC + TypeScript + Pinia + Vue Router**，复用现有 CSS 设计系统（5 套主题、密度、语气都保留）。
- **渐进式**：可先不拆 SFC，仅把 `hermes-data.js` 换成 API client 层（`api.js`），最快打通端到端；再逐步组件化。
- **鉴权**：Axios 拦截器注入 JWT、401 自动刷新、refresh 失败踢回登录页。

## 12. 安全（生产红线）

**agent 会执行代码、读写文件、联网，这是最大攻击面，必须沙箱：**

1. **每会话隔离沙箱**：每个 ACP 子进程跑在独立沙箱（**gVisor / Firecracker microVM / 容器 + seccomp + 只读根 + 独立 cwd**），用户间、团队间文件系统不互通。
2. **资源限额**：CPU/内存/磁盘/进程数 cgroups 限制；执行超时；防 fork 炸弹。
3. **网络出口控制**：默认禁外网，按需放行白名单（防数据外泄 / SSRF）。
4. **凭证隔离**：agent 自身 auth（API key / login token）按租户加密存储（Vault / KMS），spawn 时注入环境变量，**不入日志、不跨租户**。
5. **输入消毒**：消息内容、文件名、路径穿越（`fs/write_text_file` 的 path 必须 chroot 到会话目录）严格校验。
6. **常规 Web 安全**：CSRF（SameSite cookie 或双提交）、CSP、密码策略、JWT 短时效、审计全覆盖、速率限制（Redis）、依赖扫描。
7. **多租户数据隔离**：所有查询强制带 `owner_id / team_id` 作用域（service 层统一注入），防越权读。

## 13. 部署方案

### 13.1 Docker Compose（中小规模 / 起步）
```yaml
services:
  nginx:         # TLS 终止 + 静态前端 + 反代
  api:           # FastAPI（REST），可多副本
  realtime:      # FastAPI（WS/SSE）
  agent-runner:  # ACP 网关（特权受控，预装各 CLI），多副本
  worker:        # Arq/Celery 异步任务
  postgres:      # 持久卷 + 定时备份
  redis:         # AOF 持久化
  minio:         # 对象存储
  # 可选：prometheus / grafana / loki
```
- `agent-runner.Dockerfile` 预装 `claude`/`gemini`/`qwen`/`codex`/`openclaw`… CLI，启动即扫描注册。
- secrets 走 Docker secrets / `.env`（生产用外部 KMS）。

### 13.2 生产拓扑（规模化 / K8s）
- `api`/`realtime`/`worker` 无状态 Deployment + HPA。
- `agent-runner` 用 **StatefulSet / 专用节点池**（需更高隔离与本地磁盘），结合 gVisor RuntimeClass。
- PostgreSQL 用云托管（RDS）或 Patroni 高可用；Redis 用 Sentinel/Cluster。
- CI/CD：GitHub Actions → 镜像 → 灰度发布；Alembic 迁移作为发布前置步骤。
