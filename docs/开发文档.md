# 开发文档

面向在本项目上开发的工程师。讲清楚环境、结构、分层约定，以及「如何加一个接口 / 模型 / 迁移」等常见任务。

## 1. 技术栈

| 层 | 技术 |
|---|---|
| 后端 Web | FastAPI · Uvicorn（+ Gunicorn 多 worker） |
| ORM / 迁移 | SQLAlchemy 2.0（async）· Alembic |
| 校验 | Pydantic v2 · pydantic-settings |
| 数据 | PostgreSQL 16 · Redis 7 · MinIO（S3 兼容） |

## 3. 启动方式

### 3.1 一键（Docker Compose，推荐）
```bash
cp .env.example .env
make up        # 构建并启动全部服务
make logs      # 跟踪 API 日志
make down      # 停止
make fresh     # 停止并清空数据卷
```

### 3.2 本地裸机启动（无 Docker）

**后端**
```bash
cd backend
python -m venv .venv && source .venv/bin/activate
pip install -e .                      # 或 pip install -e ".[dev]" 带测试依赖
cp .env.example .env                  # 指向本机 PG/Redis
export DATABASE_URL=postgresql+asyncpg://hermes:hermes@localhost:5432/hermes
export REDIS_URL=redis://localhost:6379/0
export SECRET_KEY=dev-secret-please-change
alembic upgrade head                  # 建表
python -m app.seed                    # 建超级管理员
uvicorn app.main:app --reload         # http://localhost:8000

# Agent Runner（另开一个终端，同样的 env）
python -m agent_runner.runner         # 扫描 hermes CLI；没装则用内置 mock
```

**前端**
```bash
cd frontend
npm install
npm run dev                           # http://localhost:5173 （/api 自动代理到 :8000，含 WebSocket）
```

## 4. 后端架构与分层

请求自上而下穿过四层，**职责严格单向**：

```
HTTP/WS  →  app/api/v1/*.py      路由：解析入参、鉴权依赖、调 service、序列化出参（薄）
            app/services/*.py    业务逻辑（厚）：编排、事务、领域规则
            app/db/models/*.py   ORM 模型（SQLAlchemy 2.0）
            PostgreSQL / Redis
```

横切能力在 `app/core/`：`security`（密码/JWT）、`rbac`（平台角色）、`governance`（团队内容权限矩阵）、`redis`（连接 + Stream/PubSub/限流键）、`ratelimit`、`metrics`、`object_storage`、`logging`。

**Agent Runner**（`agent_runner/`）是独立进程，不属于 Web 请求链路：
```
runner.py        消费 Redis Stream，编排单聊 / 圆桌，写回 DB，发 PubSub 事件
acp_client.py    ACP JSON-RPC over stdio 客户端（initialize/session·new/prompt/cancel + fs/write）
discovery.py     扫描 PATH 上的 agent CLI；未找到回退 mock
session_pool.py  每会话一个长驻 ACP 子进程
sandbox.py       子进程 rlimits + 可选隔离包装
storage.py       工作区产物落库 / MinIO
mock_agent.py    内置 mock ACP 服务端（开发/测试兜底，支持 persona）
```

### 4.1 异步 SQLAlchemy 关键注意点
- 所有 DB 调用 `await`；session 来自 `Depends(get_db)`（请求级）或 `async_session_maker()`（runner / 后台）。
- **不要在响应序列化阶段触发懒加载关系**——会报 `MissingGreenlet`。已加载的关系才能序列化；需要时显式查询并手动组装 DTO（见 `conversations.py` 里 `ConversationDetail` 的拼装方式）。
- 多事件循环陷阱：模块级全局 engine/redis 的连接池绑定到首次使用的事件循环。测试用 `tests/conftest.py` 的 autouse fixture 每个用例重置 engine/redis（生产单循环无此问题）。

### 4.2 配置
全部配置走环境变量，集中在 `app/config.py` 的 `Settings`（pydantic-settings）。新增配置项就加一个带默认值的字段；通过 `from app.config import settings` 使用。完整清单见 [运维文档 §配置参考](运维文档.md#4-配置参考)。

### 4.3 Redis 用法约定（`app/core/redis.py`）
- Stream `acp:prompt`：API → runner 投递提示任务（`enqueue_prompt`）。
- PubSub `chan:conv:{id}`：runner → API 推流式事件（`publish_event`），SSE/WS 订阅转发。
- 限流键 `rl:msg:{user}`、配置键 `cfg:rate_limit_per_min`、取消键 `acp:cancel:{conv}`、JWT 黑名单 `jwt:blacklist:{jti}`。

## 5. 常见任务 Cookbook

### 5.1 新增一个 REST 接口
1. 在 `app/schemas/<域>.py` 定义请求/响应 DTO（Pydantic）。
2. 在 `app/services/<域>_service.py` 写业务逻辑（拿 `AsyncSession`）。
3. 在 `app/api/v1/<域>.py` 写路由，用 `Depends(get_current_user)` 鉴权、`Depends(require_admin())` 限管理员、`team_service.require_permission(...)` 限团队权限。
4. 若是新模块，在 `app/api/v1/__init__.py` `include_router`。
5. 加一个 `tests/test_*.py` 集成测试（见 [测试文档](测试文档.md)）。

### 5.2 新增一张表 / 改字段（迁移）
1. 在 `app/db/models/<域>.py` 加/改 ORM 模型（继承 `UUIDPrimaryKey` / `Timestamps` mixin）。
2. 在 `app/db/models/__init__.py` 导入它（让 Alembic 看到）。
3. 写迁移：
   ```bash
   cd backend
   alembic revision -m "describe change"        # 生成空模板，手写 upgrade/downgrade
   # （本项目迁移均手写以保证可控；文件名形如 000N_xxx.py，注意 down_revision 串联）
   alembic upgrade head
   ```
4. 需要 seed 数据时，在迁移里 `op.execute(...)`（注意 JSONB 用 `CAST(:d AS jsonb)` 避免双重编码，见 `0005_admin.py`）。

### 5.3 加一个团队内容权限点
在 `app/core/governance.py` 的 `PERMISSIONS` 加一项，更新 `_DEFAULTS`，在接口处用 `team_service.require_permission(db, team_id, user_id, "your.perm")` 守卫。前端权限矩阵自动渲染（来自 `/teams/{id}/policy`）。

### 5.4 加一个 agent CLI（探测）
在 `agent_runner/discovery.py` 的 `KNOWN_AGENTS` 加 `{id, bin, acp_args, ...}`，runner 重启即扫描。ACP 协议细节见 [真机联调](真机联调.md) 与 `acp_client.py`。

### 5.5 调试 Agent / ACP
- 单独跑 mock：`printf '{"jsonrpc":"2.0","id":1,"method":"initialize","params":{"protocolVersion":1}}\n' | python agent_runner/mock_agent.py`
- runner 把子进程 stderr 末尾附在错误里（`subprocess closed — stderr: ...`）；`DEBUG=true` 时 stderr 逐行打到日志。

## 6. 前端开发

### 6.1 结构
```
src/
├── api/          client.ts(axios + JWT 拦截器 + 401 自动刷新) · auth/agents/conversations/admin.ts
├── stores/       Pinia：auth.ts（会话/守卫） · chat.ts（会话/SSE/圆桌 WS）
├── router/       路由 + beforeEach 守卫（公开/登录/管理员）
├── views/        LoginView · ChatView · HistoryView · AdminView
├── components/   WorkspacePanel.vue（md/csv/json/docx 预览）
├── utils/        markdown.ts（零依赖 MD 渲染）
├── types/        全量 TS 类型
└── styles/tokens.css   设计令牌（取自原型）
```

### 6.2 约定
- **鉴权**：`api/client.ts` 请求拦截器注入 `Authorization: Bearer`，响应 401 单飞刷新；刷新失败派发 `hermes:logout` 事件 → 路由跳登录。
- **流式**：单聊用 **SSE**（`EventSource`，token 走 query 参数，因为 EventSource 不能设 header）；圆桌用 **WebSocket**（双向，`{action:'send'|'cancel'}`）。统一在 `stores/chat.ts`。
- **状态**：所有服务端状态进 Pinia store，不在组件里散落。
- **类型**：严格 TS（`strict` + `noUnusedLocals`）。构建即类型检查：`npm run build`（`vue-tsc --noEmit && vite build`）。

### 6.3 加一个页面
1. `src/api/<域>.ts` 加 API 方法（用 `http`）。
2. `src/types/index.ts` 加类型。
3. `src/views/XxxView.vue` 写视图；需要全局状态则放 `src/stores/`。
4. `src/router/index.ts` 注册路由（`meta.requiresAdmin` 可限管理员）。

## 7. 代码规范与提交

- **Python**：ruff（`line-length=100`，`pyproject.toml`）。分层不串味（路由别直接写 SQL）。函数/变量小写下划线，类型注解齐全。
- **TS/Vue**：严格类型；组件 `<script setup lang="ts">`。
- **提交**：一句话主题 + 要点列表；本仓库每个阶段（P0–P5）一个内聚提交，便于回溯。
- **测试**：改了行为就配套加/改集成测试，且要能重复跑稳定（注意共享 DB 的数据累积，用唯一标识或清理）。

## 8. 排错速查（开发期）

| 现象 | 处理 |
|---|---|
| `MissingGreenlet` | 序列化阶段触发了懒加载关系；改为显式查询 + 手动组 DTO |
| `attached to a different loop` | 测试跨事件循环复用全局池；确认 `tests/conftest.py` 生效 |
| 迁移报 JSONB 类型错 | seed JSONB 用 `CAST(:d AS jsonb)` + `json.dumps(...)`，勿对已序列化字符串再加 `type_=JSONB` |
| `/agents` 为空 | agent-runner 未起或未注册；看 runner 日志「Registered N agent(s)」 |
| 前端构建报 unused | `noUnusedLocals` 开启，清理未用变量/导入 |
